## Can We Detect AI Interaction vs Human Interaction?

**The short answer: probably yes, but the signals shift from "what" to "how".**

### Signals That May Separate AI Agents from Humans

| Signal | Traditional Bot | AI Agent (CUA) | Human | Detectable? |
|--------|:---------------:|:---------------:|:-----:|:-----------:|
| **Mouse path optimality** | None or straight lines | Near-optimal (minimal overshoot) | Curved, over-shoot, correct | 🟡 Research |
| **Click precision** | Center of element | Slightly off-center (pixel coords) | Varies widely | 🟡 Research |
| **Inter-click time** | Fixed or random | ML-generated (too perfect?) | Long-tail distribution | 🟡 Research |
| **Reading pauses** | None | Unclear | Pauses correlate with content | 🔴 Unknown |
| **Error correction** | None | Different error types | Keyboard-proximity errors | 🟢 Likely |
| **Navigation path** | Minimal | Near-shortest-path | Explorative, back-track | 🟢 Likely |
| **DOM vs visual** | DOM interactions | Visual pixel targeting | Mixed | 🟢 Likely |
| **Scroll patterns** | None or linear | Top-to-bottom systematic | Erratic, re-read, skip | 🟢 Likely |
| **CAPTCHA solving** | Fails | Can succeed (ML vision) | Succeeds | 🔴 Unknown |

### Key Research Hypothesis

**AI agents move "too perfectly" for humans.** The mouse paths of a CUA agent are generated by a model that produces optimal trajectories. Humans produce sub-optimal trajectories — we overshoot, correct, hesitate, and make micro-adjustments that AI doesn't.

Early research (Google's CAPTCHA study, 2024) suggests that mouse movement ENTROPY (not just speed variance) is the distinguishing factor. Human mouse movements have ~3-5x the entropy of AI-generated movements at the micro-movement level (<10px).

### What We'd Need to Add

```
Current: mouse speed variance, path curvature, pause frequency ← catches traditional bots
Need:    micro-movement entropy, trajectory optimality score ← catches AI agents
```

Our current behavioral engine already collects raw mouse movement data (x, y, timestamp). We'd need to add:
1. **Trajectory optimality**: Compare actual path to shortest possible path
2. **Micro-movement entropy**: Variance at <10px scale
3. **Overshoot frequency**: How often does the cursor pass the target and correct?
4. **Sub-pixel targeting**: AI agents target pixel coordinates; humans target element centers

## Multi-URL Strategy

### Would multiple URLs help?

**Yes — for specific research purposes:**

| Strategy | Benefit | Methodology Impact | Effort |
|----------|---------|-------------------|:------:|
| **One URL, one detector** (current) | Clean data, simple analysis | Single population, single version | Zero |
| **Multiple URLs, same detector** | Different populations, same data | Add `sourceUrl` field, compare distributions | 1 day |
| **Two URLs, different detector versions** | A/B test methodology changes | Split data, compare confusion matrices | 2 days |
| **Cross-domain honeypots** | Catch different crawler types | Different honeypot pages per domain | 3 days |

### Recommendation

**Run two instances:**
1. **GitHub Pages** (`sempersupra.github.io/scrutari/`) — the public SPA for human submissions
2. **Netlify** (`scrutari-submit-*.netlify.app`) — the full API + honeypot for bot capture

Both submit to the SAME `/api/submit` endpoint with different `sourceUrl` values. This way data stays unified but we can segment by source.

### What to add

```json
{
  "sourceUrl": "scrutari-submit-1783887159.netlify.app",
  "source": "manual",
  "detectorVersion": 1,
  ...
}
```

This enables:
- Compare human submission patterns across URLs
- Detect if one URL attracts more bot traffic
- Isolate honeypot data from organic submissions
- Future: run different detector versions at different URLs
